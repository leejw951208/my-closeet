# Spec. 옷 등록 (item-register)

> 한 줄 요약. 카메라 또는 갤러리에서 옷 사진을 받아 AI 자동 분류와 함께 8초 안에 내 옷장에 등록한다.

## 배경

My Closet MVP의 진입 가치는 "내 옷을 디지털 옷장에 빨리·정확히 넣을 수 있는가"다. 옷이 등록되지 않으면 옷장 탐색·코디·캘린더·공유 어느 것도 의미가 없다.

지금은 사용자가 옷장을 만들기 위해 별도 도구(사진 앱·메모·외부 코디 앱)로 사진을 모아 수동 분류해야 한다. 한 벌 등록에 1분 이상 걸리고, 카테고리·색상을 직접 입력해야 한다. 50벌을 채우는 데만 50분이 들어 시작 단계에서 이탈한다.

이 슬러그는 (1) 카메라/갤러리 단건 등록, (2) 갤러리 최대 20장 일괄 등록, (3) 배경 제거·카테고리 자동 분류, (4) 1탭 카테고리 수정의 4개 능력을 묶어 "사진을 찍거나 고르기만 하면 옷장에 들어가는" 경험을 제공한다. PRD US-02·US-03·US-04를 한 슬러그로 수직 슬라이스한다.

## 기능 목록

### F1. 단건 옷 등록 (카메라·갤러리)

사용자가 카메라로 옷을 찍거나 갤러리에서 한 장을 선택해 옷장에 추가하는 가장 기본 흐름.

**동작 방식**

- 옷장 화면 우하단 "+" → 카메라/갤러리 선택 시트.
- 카메라 권한 미허용 시 OS 권한 다이얼로그, 거부 시 설정 진입 유도.
- 촬영·선택 후 자동 배경 제거가 적용된 미리보기 노출, 1탭으로 "원본 사용"·"다시 찍기" 전환.
- "저장" 탭 시 사전서명 URL로 클라이언트가 R2/MinIO에 직접 업로드, 메타데이터(closetId·photoUrl·category·colorHex·aiConfidence)를 서버에 기록.
- 업로드 실패 시 재시도 큐에 적재, 백그라운드 재시도.

**포함 범위**

- iOS·Android 카메라/갤러리 권한 처리.
- 클라이언트 측 이미지 압축(긴 변 1600px·JPEG quality 85).
- 사전서명 URL 발급(`POST /items/presign`)·메타 저장(`POST /items`).
- 등록 평균 소요 ≤8초(카메라 진입~저장 완료, 실기기 3종 측정).

**제외 범위**

- 옷별 수동 이모지 지정. → `closet-browse` D2에서 처리.
- 태그·브랜드·계절 수기 입력. → B4(별도 슬러그·후속) 범위.

### F2. 갤러리 일괄 등록 (최대 20장)

신규 사용자가 옷장 초기 시드를 빠르게 채울 수 있게 갤러리에서 다중 선택 → 백그라운드 일괄 처리.

**동작 방식**

- 갤러리 시트에서 최대 20장 다중 선택.
- 클라이언트가 각 사진을 순차 압축 → `POST /items/batch`로 batchId 생성 → 항목별 사전서명 URL 발급 → 백그라운드 큐가 순차 업로드·분류.
- 진행률 표시("3/20"), 사용자는 다른 화면 이동 가능.
- 처리 완료 시 인앱 토스트, 실패 항목은 별도 "실패 목록"에서 재시도 버튼 제공.

**포함 범위**

- 20장 동시 다중 선택 UI.
- 백그라운드 큐(앱 백그라운드 진입 시에도 진행, OS 제약 내에서).
- 진행률 폴링 엔드포인트(`GET /items/batch/:batchId`).

**제외 범위**

- 20장 초과. → 두 번 나눠 등록 안내.
- 푸시 알림. → MVP에서 인앱 토스트만, OS 푸시는 v1.5.

### F3. AI 자동 분류 (배경 제거 + 카테고리 Top-3)

업로드된 옷 사진을 받아 배경 제거·카테고리 Top-3·대표 색상을 추론해 등록 메타데이터를 자동 채운다.

**동작 방식**

- API 게이트웨이가 객체 키를 받아 분류 마이크로서비스(`POST /classify`)에 위임.
- 분류 서비스는 (1) 배경 제거 결과 이미지를 새 객체 키로 저장, (2) 카테고리 Top-3 라벨·신뢰도, (3) 대표 색상 HEX를 반환.
- 응답 Top-1을 Item.category에, 신뢰도를 Item.aiConfidence에, 색상을 Item.colorHex에 저장.
- Top-1 신뢰도 < 0.5 → category = null 로 두고 "미분류" 표시, 사용자에게 직접 선택 요청.

**포함 범위**

- 8개 카테고리(TOP·BOTTOM·DRESS·OUTER·SHOES·HAT·BAG·ACCESSORY) 분류 모델 v1.
- 회귀 셋 100장 Top-1 ≥80%·Top-3 ≥95% 검증.
- 모델 버전·신뢰도·요청 ID 로깅.

**제외 범위**

- 옷의 패턴·소재·핏 인식. → v1.5 이후.
- 사용자 보정 데이터 기반 재학습 자동 파이프라인. → 익명 수집만 하고 학습은 수동.

### F4. 카테고리 1탭 수정

AI 분류 결과가 틀렸을 때 사용자가 1탭 거리에서 Top-3 후보 또는 전체 목록에서 수정할 수 있게 한다.

**동작 방식**

- 등록 완료 화면 또는 옷 상세 화면에서 카테고리 칩 옆 "수정" 아이콘.
- 탭 시 Top-3 후보가 큰 버튼으로 노출, "더 보기" → 8개 전체 목록.
- 선택 즉시 `PATCH /items/:id/category` 호출, 별도 저장 버튼 없음.
- 보정 이벤트가 분석 파이프라인에 기록(원래 카테고리·수정 카테고리·신뢰도).

**포함 범위**

- 단건 수정 UX·API.
- 보정률(corrected/total) 분석 이벤트.

**제외 범위**

- 색상 수정·태그 입력. → B4(후속).
- 일괄 수정. → MVP 범위 아님.

## 입출력

**입력**

- `POST /items/presign`. `{ contentType: "image/jpeg"|"image/png" }`. 인증 필수.
- `POST /items`. `{ objectKey: string, contentTypeHash?: string }`. 인증 필수. objectKey는 presign 응답 그대로.
- `POST /items/batch`. `{ count: number(1~20), contentType: string }`. 인증 필수. `{ batchId, items: [{ itemId, uploadUrl, objectKey }] }` 반환.
- `GET /items/batch/:batchId`. 인증 필수. 진행률·실패 항목 반환.
- `PATCH /items/:id/category`. `{ category: ItemCategory }`. 인증 필수.

**출력**

- `POST /items/presign`. `{ uploadUrl, publicUrl, objectKey, expiresInSeconds }`.
- `POST /items`. `{ id, category, colorHex, photoUrl, aiConfidence, registeredAt }`. 동기 응답 안에 분류 결과 포함(타임아웃 2초 초과 시 category=null로 즉시 응답하고 백그라운드 재분류).
- `POST /items/batch`. 위 참조.
- `GET /items/batch/:batchId`. `{ batchId, total, completed, failed, items: [{ itemId, status, error? }] }`.
- `PATCH /items/:id/category`. `{ id, category, updatedAt }`.

## 제약 조건

- 사진 단건 최대 10MB(서버 측 거부), 클라이언트 압축 후 평균 0.5~1.5MB.
- 배치는 한 사용자 동시 1개. 진행 중 새 배치 요청 시 409.
- 분류 마이크로서비스 P95 ≤2초(3G 제외), 타임아웃 5초.
- 미분류(category=null) Item도 옷장 그리드에는 노출, "미분류" 섹션에 모음.
- 사진은 본인만 접근. presigned URL 1시간 만료. 객체 키는 `users/{userId}/items/{itemId}.jpg` 규칙.

## 예외 케이스

- 카메라/저장소 권한 거부 → 권한 안내 화면 + 설정 진입 버튼.
- 네트워크 단절 → 업로드 큐에 적재, 재진입 시 자동 재시도. 사용자에겐 "대기 중" 배지.
- 사전서명 URL 발급 성공·R2 업로드 실패(만료·5xx) → 클라이언트 1회 재발급·재시도, 2회 연속 실패 시 실패 목록.
- 분류 서비스 타임아웃(>5초) → Item.category = null, aiConfidence = null로 저장하고 백그라운드 재분류 작업 enqueue.
- 분류 신뢰도 < 0.5 → category = null, 사용자에게 직접 선택 요청.
- 배치 중 일부 실패 → 성공분은 옷장에 들어가고, 실패분은 `GET /items/batch/:batchId` 응답의 failed 항목으로 노출.
- 동일 batchId 중복 요청 → idempotent 처리(첫 응답과 동일 반환).
- 20장 초과 선택 → 클라이언트에서 사전 차단·안내 토스트.

## 채택 근거

**핵심 이유**

- "등록이 빨라야 옷장이 채워지고, 옷장이 채워져야 코디 가치가 나온다." 등록 마찰을 8초까지 줄이지 못하면 MVP 가설(D7 ≥30%) 자체가 무너진다.

**보조 이유**

- AI 자동 분류는 수동 입력 노동을 제거해 신규 사용자가 50벌을 10분 안에 채울 수 있게 한다. PoC 검증된 모델 라이브러리(예. MediaPipe/CLIP 기반)를 v1으로 사용해 학습 비용 없이 빠르게 출시.
- 사전서명 URL 방식은 API 서버가 사진 바이트를 거치지 않게 해 비용·지연을 동시에 낮춘다.

**기각된 대안**

- **클라이언트 분류 모델 온디바이스.** 모델 사이즈·정확도·기기 편차 부담. 1인 운영 단계에서 서버 호출이 단순.
- **수동 카테고리 입력만.** 등록 마찰이 분당 1~2벌로 떨어져 옷장 시드 채우기에서 이탈. PRD US-04 정확도 게이트로 자동 분류 필수.
- **단건 API 하나로 batch까지 처리.** 진행률·재시도·idempotency를 단건과 한 묶음으로 처리하면 복잡도 증가. 별도 batch 엔드포인트로 분리.

## 비기능 요건

**성능**

- 등록 평균(카메라 진입~저장 완료). ≤8초(실기기 3종 평균).
- `POST /items` 동기 응답. P95 ≤2.5초(분류 포함). 분류 타임아웃 초과 시 즉시 응답.
- `POST /classify`. P95 ≤2초, 동시 1,000 요청에서 에러율 ≤0.5%.
- 배치 20장 처리. 평균 ≤90초(순차 처리 가정).

**보안**

- 위협 모델. 본인 옷 사진의 무단 열람 방지. 보호 범위는 객체 스토리지·메타 DB·API. 범위 외는 사용자가 OS·다른 앱에서 공유한 후의 흐름.
- 모든 객체 키는 userId를 경로에 포함, presigned URL은 1시간 만료.
- 메타 API 모두 JWT 인증·소유자 검증(`closet.userId === currentUser.id`).
- 사진 URL은 항상 사전서명, 영구 공개 URL 발급 금지.

**확장성**

- 가정 규모. MAU 1만 명·1인당 평균 80벌. 총 80만 Item, 사진 0.8TB.
- 그 이상은. R2 비용·분류 서비스 GPU 비용 모니터링 후 (1) 분류 마이크로서비스 수평 확장, (2) 사진 lifecycle 정책(180일 콜드)으로 대응.

## 용어 정의

- **Closet.** 사용자 1인당 1개 보유하는 옷장 컨테이너. User와 1:1.
- **Item.** 옷 한 벌. Closet에 속하며 사진·카테고리·색상·신뢰도를 갖는다.
- **batchId.** 일괄 등록 한 묶음을 식별하는 UUID. 진행률 조회 키.
- **사전서명 URL(presigned URL).** 객체 스토리지가 발급한 단기 PUT 허용 URL. 클라이언트가 직접 사진을 올릴 수 있게 한다.
- **미분류.** AI 신뢰도가 임계치 미만이거나 분류 실패한 Item의 상태. `category = null`로 표현.
- **보정률.** AI Top-1과 다른 카테고리로 사용자가 수정한 비율. 분석 KPI.
